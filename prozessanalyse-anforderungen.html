Prozessanalyse & Anforderungsableitung

Diese Arbeitsprobe zeigt, wie aus einem fachlichen Ablauf ein strukturierter Prozess, erkennbare Schwachstellen, konkrete Anforderungen, User Stories, Akzeptanzkriterien und eine kurze Standard-Arbeitsanweisung entstehen können.

Der Schwerpunkt liegt bewusst nicht auf Programmierung, sondern auf der Analyse, Strukturierung und verständlichen Dokumentation an der Schnittstelle zwischen Fachbereich, Organisation und IT.

1. Ausgangssituation

In einem Web- oder Redaktionsteam werden regelmäßig Informationen für Veranstaltungen, Beiträge oder organisatorische Hinweise gesammelt, geprüft und veröffentlicht. Der Ablauf wirkt auf den ersten Blick einfach, kann aber durch unklare Zuständigkeiten, fehlende Pflichtangaben oder späte Freigaben schnell unübersichtlich werden.

Typische Kommunikationswege sind E-Mail, Messenger, persönliche Absprachen oder mündliche Hinweise. Dadurch entstehen Rückfragen, Verzögerungen und zusätzlicher Abstimmungsaufwand.

2. Ziel der Arbeitsprobe

Ziel dieser Arbeitsprobe ist es, einen typischen Veröffentlichungsprozess strukturiert zu analysieren, Schwachstellen sichtbar zu machen und daraus fachliche sowie technische Anforderungen abzuleiten.

Der Fokus liegt auf einer nachvollziehbaren Arbeitsweise: vom Ist-Prozess über erkannte Probleme bis hin zu Anforderungen, User Stories, Akzeptanzkriterien und einer kompakten SOP.

3. Ist-Prozess

  1. Eine Gruppe oder verantwortliche Person möchte einen Inhalt veröffentlichen.
  2. Informationen werden per E-Mail, Messenger oder mündlich weitergegeben.
  3. Das Web- oder Redaktionsteam prüft, ob alle notwendigen Angaben vorhanden sind.
  4. Bei fehlenden Informationen entstehen Rückfragen.
  5. Ein Text wird erstellt, angepasst oder gekürzt.
  6. Eine Freigabe wird eingeholt.
  7. Der Beitrag wird veröffentlicht.
  8. Nachträgliche Korrekturen werden erneut manuell gemeldet.

4. Schwachstellenanalyse

  • Pflichtinformationen fehlen häufig oder werden uneinheitlich geliefert.
  • Zuständigkeiten sind nicht immer eindeutig.
  • Freigaben erfolgen teilweise spät oder informell.
  • Änderungen sind nicht durchgängig nachvollziehbar.
  • Es gibt keine zentrale Übersicht über offene Veröffentlichungen.
  • Fristen werden nicht immer klar dokumentiert.
  • Qualitätssicherung hängt stark von Einzelpersonen ab.

5. Soll-Prozess

  1. Eine Veröffentlichungsanfrage wird über ein standardisiertes Formular erfasst.
  2. Pflichtfelder stellen sicher, dass alle relevanten Informationen vorliegen.
  3. Verantwortliche Person, Zielgruppe und Veröffentlichungsfrist werden festgelegt.
  4. Das Web- oder Redaktionsteam prüft Inhalt, Format und Vollständigkeit.
  5. Falls nötig, werden Rückfragen gebündelt gestellt.
  6. Die Freigabe wird dokumentiert.
  7. Die Veröffentlichung erfolgt nach einer Checkliste.
  8. Nach Veröffentlichung wird eine kurze Nachkontrolle durchgeführt.

6. Abgeleitete Anforderungen

Anforderung

A-01: Strukturierte Erfassung

Das System soll Veröffentlichungsanfragen strukturiert erfassen können.

Anforderung

A-02: Pflichtfelder

Das System soll Pflichtfelder für Titel, Beschreibung, Datum, Zielgruppe, verantwortliche Person und Veröffentlichungsfrist enthalten.

Anforderung

A-03: Statusübersicht

Das System soll den Bearbeitungsstatus einer Anfrage sichtbar machen.

Anforderung

A-04: Freigabe dokumentieren

Das System soll dokumentieren können, ob eine Freigabe erfolgt ist.

Anforderung

A-05: Veröffentlichungscheckliste

Das System soll eine Checkliste für die Prüfung vor Veröffentlichung bereitstellen.

Anforderung

A-06: Nachkontrolle

Das System soll nach Veröffentlichung eine kurze Nachkontrolle ermöglichen.

7. User Stories

User Story

Strukturierte Anfrage

Als Mitglied eines Fachbereichs möchte ich eine Veröffentlichungsanfrage strukturiert einreichen, damit alle notwendigen Informationen direkt vollständig vorliegen.

User Story

Status erkennen

Als Webteam-Mitglied möchte ich den Status einer Anfrage sehen, damit ich offene Aufgaben besser priorisieren kann.

User Story

Freigabe prüfen

Als freigebende Person möchte ich den finalen Text prüfen können, damit nur abgestimmte Inhalte veröffentlicht werden.

User Story

Übersicht behalten

Als teamverantwortliche Person möchte ich eine Übersicht über offene Veröffentlichungen haben, damit Fristen und Zuständigkeiten transparent bleiben.

8. Akzeptanzkriterien

Beispielhafte Akzeptanzkriterien für die Anforderung A-01: Strukturierte Erfassung:

  • Eine neue Anfrage kann mit Titel, Beschreibung, Datum, Zielgruppe und Frist angelegt werden.
  • Pflichtfelder müssen ausgefüllt sein, bevor die Anfrage gespeichert werden kann.
  • Nach dem Speichern erhält die Anfrage automatisch den Status „Neu“.
  • Die Anfrage ist in einer zentralen Übersicht sichtbar.
  • Fehlende Pflichtangaben werden verständlich angezeigt.

9. Mini-SOP: Veröffentlichungsanfrage bearbeiten

SOP bedeutet Standard Operating Procedure. Gemeint ist eine standardisierte Arbeitsanweisung, die beschreibt, wie ein wiederkehrender Vorgang zuverlässig durchgeführt wird.

Zweck

Sicherstellen, dass Veröffentlichungsanfragen vollständig, geprüft, freigegeben und nachvollziehbar veröffentlicht werden.

Rollen

  • Fachbereich oder anfragende Person: liefert Inhalt und Pflichtangaben.
  • Web- oder Redaktionsteam: prüft, bearbeitet und veröffentlicht den Inhalt.
  • Freigebende Person: prüft und bestätigt die Veröffentlichung.

Ablauf

  1. Anfrage erfassen.
  2. Pflichtangaben prüfen.
  3. Rückfragen klären.
  4. Text vorbereiten.
  5. Freigabe einholen.
  6. Veröffentlichung durchführen.
  7. Nachkontrolle durchführen.

Definition of Done

  • Alle Pflichtangaben liegen vor.
  • Freigabe ist dokumentiert.
  • Beitrag ist veröffentlicht.
  • Darstellung wurde nach Veröffentlichung geprüft.
